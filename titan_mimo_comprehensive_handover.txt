## 1. EXECUTIVE SUMMARY
The **Titan MIMO 650M** is a ~650 million parameter language model built entirely from scratch in PyTorch. It represents a radical departure from traditional Transformer architectures and standard optimization techniques. 
1.  **Architecture:** It utilizes the Mamba S6 (Selective State Space) algorithm rather than attention, allowing linear-time sequence processing. Furthermore, it employs a Multi-Input Multi-Output (MIMO) Mixture-of-Experts architecture, branching the backbone into four distinct expert arms (Chat, Math, Code, Tool).
2.  **Optimization:** The network completely abandons FP32 continuous gradient descent (no AdamW/SGD) for its core linear projections. Instead, it utilizes the **PRIME Optimizer**, a purely discrete, vote-based mechanism that maps weights to a static Prime Harmonic Look-Up Table (LUT).

---

## 2. THE PRIME OPTIMIZER (ContinuousVoteOptimizer)
The core `in_proj` and `out_proj` weight matrices of every Mamba layer are wrapped in a custom `PrimeLinear` module. These matrices do not exist as standard FP32 continuous parameters. 

**Mechanics:**
*   **Prime Harmonic LUT:** A shared 65,536-element float32 Look-Up Table (LUT) populated with prime harmonic fractions (e.g., 1/2, 1/3, 1/5) and interpolated edges.
*   **Discrete Pointers:** Every weight is stored as a `uint16` index pointing to a specific value in the LUT. 
*   **Vote Accumulation:** During the backward pass, gradients are converted strictly to their sign (+1 or -1) and multiplied by 10. These integer "pressures" accumulate inside an `int16` `vote_buffer` across micro-batches.
*   **Supermajority Gate:** A parameter is only permitted to "flip" (shift its pointer up or down the LUT by 1 index) if its accumulated vote pressure exceeds a dynamic threshold. 
*   **GPU VRAM Optimization:** All indices, buffers, and voting logic are registered as native GPU tensors, eliminating CPU/GPU transfer bottlenecks and allowing training throughput to reach ~660 Tokens/Sec on an RTX 3060.

---

## 3. AUTONOMOUS PI CONTROLLER (Homeostatic Volatility)
In a 650M parameter model, layers experience vastly different gradient variances. A static supermajority threshold caused early layers to freeze (0% flips) and late layers to become stochastic noise generators (30%+ flips), destroying convergence.

**The Fix:**
Every single `PrimeLinear` layer possesses an autonomous Proportional-Integral (PI) Controller operating in log-space.
*   The user sets a `target_flip_rate` (e.g., 10% for backbone layers, 16% for the domain router).
*   Every 50 steps, the PI controller calculates the layer's Exponential Moving Average (EMA) flip rate.
*   It compares the EMA to the target rate and dynamically adjusts the supermajority `log_threshold` (using $K_p=0.5, K_i=0.01$). 
*   If a layer flips too aggressively, the PI controller raises the threshold, mathematically forcing the layer to require stronger gradient consensus before altering weights. This absorbs massive gradient shocks without catastrophic forgetting.

## 4. TITAN MIMO ARCHITECTURE
To prevent "catastrophic forgetting" across widely different domains (e.g., Python syntax overriding Conversational English syntax), the model is bisected.

*   **Continuous Shell:** Token embeddings, LayerNorms, and the final LM Head remain continuous FP32 to allow smooth vocabulary mapping.
*   **Shared Trunk (Layers 0-13):** The first half of the Mamba backbone processes raw token syntax generically.
*   **Dynamic Domain Router:** At the mid-point, a linear layer analyzes the unnormalized residual stream context and autonomously decides which expert arm is required.
*   **Expert Arms (Chat, Math, Code, Tool):** The forward pass is routed into one of four distinct Mamba S6 blocks. These blocks specialize deeply in their respective latent geometries.
*   **Shared Crown (Layers 14-27):** The expert arm output is collapsed back into the residual stream and processed by the final generic reasoning layers.

## 5. TRAINING PHASE HISTORY & BUG RESOLUTION
This model has been systematically trained through 4 distinct phases to isolate variables and stabilize the highly experimental architecture.

### Phase 1 & 2: Infrastructure & Porting
*   **CPU NaNs:** The pure PyTorch Mamba S6 complex cumulative sums (e.g., `einsum`, `F.softplus`) suffered massive floating-point precision loss on standard CPU ALUs. Inference/Evaluation was strictly bound to GPU CUDA kernels to prevent NaN divergence.
*   **VRAM Constraints:** Batch Size 4 caused OOM crashes during inference generation. Training runs at Batch Size 1, Grad Accumulation 2. Evaluation scripts require explicitly pausing the training loop to clear VRAM.

### Phase 3: Domain Grokking & The Padding Bug
In this phase, the routing was strictly hard-coded based on the dataset domain tags, forcing the model to grok its separated expert arms without autonomous confusion.
*   **The Artificial Padding Collapse:** At Step 217k, the loss dropped artificially to 1.73, but the model generated hallucinatory word salad. The network had hyper-optimized its lattice to predict padding tokens (`<|endoftext|>`), which comprised 90% of short instruction batches.
*   **Fix:** `F.cross_entropy` was updated with `ignore_index=0`, instantly restoring English coherence.

### Phase 4: Dynamic Router Unfreezing (CURRENT ACTIVE STATE)
At step 313,800, the `domain_router` was unfrozen, wrapped in the PrimeDiscrete optimizer, and allowed to route autonomously. This caused severe MoE scaling anomalies that required surgical fixes:
*   **OOD Blending Explosion:** The router's initial `F.softmax` created fractional routes (e.g., 0.3 Chat, 0.5 Math). Blending these caused Out-Of-Distribution (OOD) residual states that mathematically exploded the downstream Mamba selective-scan algorithms. **Fixed by replacing Softmax with `F.gumbel_softmax(hard=True)`**, forcing strict 1-hot routes via a Straight-Through Estimator while preserving backprop gradients.
*   **Unnormalized Residual Variance:** The routing cross-entropy loss spiked to 2.6 million. The router was reading the mid-network residual stream which had a massive variance of +/- 210.0, causing explosively large logits. **Fixed by applying a detached `F.layer_norm`** to the residual state prior to routing, bounding the loss strictly to ~1.38.
*   **Inference Hallucinations & Padding Gradients:** `F.gumbel_softmax` applies noise unconditionally. **Fixed by bypassing Gumbel noise with a greedy `torch.argmax` during `model.eval()` inference**. Additionally, we masked out padding tokens in the routing loss (`ignore_index=-100`) so the router wouldn't waste gradients categorizing blank space.
